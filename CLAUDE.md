# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Embedded firmware for an Infineon AURIX **TC264DA/TC264D** (TC26B family) dual-core TriCore
microcontroller, targeting the "ÈæôÈÇ±" (Longqiu/LQIST) TC26x robotics competition mainboard. The
robot performs several competition tasks: picking/placing objects with a pump-actuated gripper
(task1), counting/picking/placing dropped water bottles (task2), radar-triggered shooting
(task3), and ball spotting (task4) ‚Äî plus line-following and IR-array-based navigation that ties
them together. It communicates with an external vision host (e.g. a Jetson Nano) over UART and
drives a set of Fashion Star bus servos for the arm.

Firmware is developed in **AURIX Development Studio (ADS) 1.6**, an Eclipse/CDT-based IDE, and
built with the **TASKING TriCore C compiler**. There is no OS ‚Äî `Cpu0_Main.c`/`Cpu1_Main.c` are
bare-metal `while(1)` loops, one per core. **Most of the actual task logic runs in interrupt
handlers, not the main loop** ‚Äî see "Real-time architecture" below before assuming a function is
unused just because the main loop doesn't call it.

## Build / flash / debug

This is an Eclipse CDT project, not a standalone CLI build. There is no test suite (embedded
firmware controlling physical hardware).

- **Build**: Open the project root in AURIX Development Studio, right-click the project ‚Üí
  `Build Project` (or `Clean Project` then `Build Project` first, to catch stale artifacts).
  A command-line build is technically possible via `Debug/makefile` (`make -C Debug all`) but
  requires the TASKING toolchain on `PATH`; ADS is the supported workflow.
- **Flash/debug**: Use the `.launch` configs at the project root (`LQ_Encoder_LIB.launch`) via
  ADS's Debug As. If a `.launch` file is stale/broken, delete it and let ADS regenerate one
  (right-click ‚Üí Debug As, or see `ÈæôÈÇ±ADS&MRSË∞ÉËØïÈóÆÈ¢òÁöÑ‰∏ÄËà¨Â§ÑÁêÜÊñπÊ≥ï.txt` for the documented
  troubleshooting steps: clean/rebuild first, delete+regenerate `.launch`, fall back to
  `memtool` for flashing if ADS download fails).
- **Path constraint**: the project directory itself must live under a short, all-ASCII path
  (e.g. `C:\LQ_TC26xB_LIB\...`) ‚Äî spaces or non-ASCII (e.g. Chinese) characters in the path are a
  known cause of TASKING build failures. This is called out in multiple files in this repo

  is checked into this repo as ADS's build output ‚Äî expect it to show as modified after every
  build. Don't hand-edit anything under `Debug/`; it's regenerated by the IDE.

## Repository layout

Two source trees are both compiled into the same binary (see `Debug/sources.mk`
`SUBDIRS`) ‚Äî don't assume `src/` is dead code:

- **`Core/`** ‚Äî the active application logic for this specific robot build: task state machines,
  motor/servo control, IR sensors, radar ("cheju") receiver, and glue code. This is where most
  feature work happens. Files here are dated by "Created on" comments (Mar‚ÄìJul 2026), i.e. this
  is the newest/actively-developed layer.
  - `Motor.c` (~1800 lines) ‚Äî the core of the robot: PID speed loop, encoder handling, and the
    per-task position/state-machine logic (`task1_...`, `task2_...`, `TASK3_...`,
    `all_task_position_loop_init`, `position_loop`, line-crossing detection, angle correction).
    Most task-specific behavior changes land here.
  - `Servo.c`/`Servo.h` ‚Äî wraps the Fashion Star bus servo driver (see below) for the gripper arm
    and defines the `dummy_struct` (pump + servo angle state) used by the pick/place tasks.
  - `cheju.c`/`cheju.h` ‚Äî UART-framed radar distance receiver (`radar1`/`radar2`,
    frame headers `0xBB`/`0xCC`), used by task1/task3 distance gating.
  - `IR_sensor.c`/`IR_sensor.h` ‚Äî reads two 6‚Äì8 pin IR sensor arrays (`sr1_pin*`, `sr2_pin*`) used
    for line following/crossing.
  - `Communication.c`/`Task.c` ‚Äî currently thin scaffolding (small `TASK_COMMAND` enum,
    mostly empty bodies); not yet where the real task logic lives (that's still in `Motor.c`).
  - `Task.h` transitively pulls in `Motor.h`+`Servo.h`, and `Motor.h`/`Servo.h` pull in each
    other's headers (`cheju.h` too) ‚Äî expect a fairly tangled include graph in `Core/`.
- **`src/`** ‚Äî a more generic Longqiu TC264 board-support layer, organized as:
  - `src/Driver/LQ_UART.c` and `src/Driver/LQ_CCU6.c` ‚Äî **not just drivers**: their interrupt
    handlers are where most of the task orchestration actually lives. See
    "Real-time architecture" below.
  - `src/Driver/` (rest) ‚Äî low-level peripheral wrappers (`LQ_GPIO`, `LQ_ADC`, `LQ_GTM` (PWM),
    `LQ_GPT12_ENC` (quadrature encoders), `LQ_SPI`/`LQ_QSPI`, software I2C/SPI, EEPROM, EMEM,
    FFT).
  - `src/APP/` ‚Äî higher-level reusable modules on top of `Driver/`: motor PWM (`LQ_Atom_Motor`),
    displays (`LQ_TFT18`, `LQ_TFT2`, `LQ_OLED096`, `LQ_IPSLCD`), IMUs (`LQ_ICM20602`,
    `LQ_LSM6DSR`, `LQ_I2C_9AX`, `LQ_IIC_Gyro`), buttons/LEDs, VL53 ToF sensor.
  - `src/User/` ‚Äî the intended extension point for user code on top of the board-support layer
    (`LQ_MotorServo.c`); mostly a stub in this repo, since bot-specific logic lives in `Core/`
    instead.
  - `src/Configuration.h` / `src/ConfigurationIsr.h` ‚Äî board/ISR configuration touched by
    generated Infineon config code.
- **`Core/fashion_star_uart_servo/`** and **`Core/fashion_star_uart_servo_examples/`** ‚Äî vendored
  driver for the Fashion Star bus servos (`FSUS_SetServoAngle`, sync servo control) used by
  `Core/Servo.c`.
- **`Core/ring_buffer/`**, **`Core/user_uart/`** ‚Äî vendored ring-buffer and UART-abstraction
  helpers layered under `LQ_UART`; `User_Uart_Init` is set up in `Cpu0_Main.c` for the servo bus
  UART.
- **`Irq/irq.c`** ‚Äî essentially an empty stub (one unused `GPT1_Timer_Init`, no ISR body). Despite
  the name, this is **not** where the CCU6/UART interrupt handlers live ‚Äî don't look here for
  runtime logic; see `src/Driver/LQ_CCU6.c` and `src/Driver/LQ_UART.c` instead.
- **`Libraries/`** ‚Äî vendored, not hand-edited:
  - `Libraries/iLLD/TC26B/...` ‚Äî Infineon low-level driver (iLLD) for TC26B peripherals
    (Gtm, Ccu6, Gpt12, Vadc, Qspi, Asclin, Stm, Scu, Cpu/CStart, Cpu/Trap, ...).
  - `Libraries/Infra/`, `Libraries/Service/` ‚Äî Infineon platform/service support code
    (register headers under `Infra/Sfr/TC26B`, generic CPU services under `Service/CpuGeneric`).
  - `Libraries/pinmapper.pincfg` ‚Äî pin mapping config for Infineon's pin mapper tool; if you
    change a peripheral pin assignment, this and the generated `_PinMap` sources should stay
    consistent.
- **`Configurations/`** ‚Äî user clock/GPIO/interrupt configuration (`Ifx_Cfg.h`,
  `Configurations/Debug/sync_on_halt.c`) generated/edited via ADS's config perspective.
- **`Cpu0_Main.c`** ‚Äî CPU0 entry point (`core0_main`): peripheral init (UARTs, LEDs, keys, IMU,
  encoders, motors, IR sensors, servos), then the main polling loop that drives task state
  transitions, updates the TFT/OLED debug display, and reads the radar.
- **`Cpu1_Main.c`** ‚Äî CPU1 entry point (`core1_main`): waits on `mutexCpu0InitIsOk` released by
  CPU0, then is effectively idle (a `TFT_WORK_ON_ONE_CPU`-gated display init exists but the
  active build keeps display ownership on CPU0 via `mutexCpu0TFTIsOk`).

## Real-time architecture (read this before tracing "who calls X")

Task logic is split across three layers by cadence, not by task number. A given task (say,
task2) has code in all three:

1. **`src/Driver/LQ_UART.c` RX interrupt handlers** ‚Äî parse framed input from external sensors
   and set the flags/targets everything else reacts to. This is the actual entry point for
   externally-driven task transitions, not the main loop:
   - `UART0_RX_IRQHandler` ‚Äî the vision-host protocol. Frame = `0xAA` + a mode byte
     (`0x01`..`0x06`) selecting a payload shape, terminated by `0x0D`: line-following deviation
     (mode 1, drives `following_speed[]` from a camera-lane-error P controller), task1
     pick/put pixel offsets + the `pick_or_put` trigger (mode 2), task3's `worm_record_array`
     target bitmap (mode 4, also clears `is_waiting_for_task_record`), task2 water-drop xy
     calibration + drop count ‚Üí `TASK2_DROP_COUNT`/`task2_current_state` (mode 3), task4 ball xy
     offset + `task4_ball_current_state` (mode 5).
   - `UART3_RX_IRQHandler` ‚Äî radar link: feeds each byte to `Radar_Feed_Byte()` (see
     `Core/cheju.c`), then calls `task2_tast_dis()`, `task4_test_distance()`,
     `task3_start_test_distance()` **on every received byte**. These three functions are declared/
     defined in `Core/cheju.c` but have no call site there or in the main loop ‚Äî this UART3 ISR is
     their only caller. They check `radar1`/`radar2.distance` + `wheel_asix.yaw` and, once in
     range, set the `*_prepare_correct`/`*_start_test_distance` flags that the CCU6 layer and main
     loop pick up.
2. **`src/Driver/LQ_CCU6.c` timer interrupt handlers** ‚Äî the actual periodic task *scheduler*:
   - `CCU60_CH0_IRQHandler` (5ms, set up via `CCU6_InitConfig(CCU60, CCU6_Channel0, 5000)` in
     `Cpu0_Main.c`) is a long if/else chain that, based on which flag is currently set
     (`following_flow_start`, `is_task1_wheels_moving_to_next_point`,
     `is_task1_wheels_moving_to_last_point`, `task2_start_correct`, `tast3_start_to_correct`,
     `does_start_moving_three_water_in_one/two`, `task3_go_back_to_in_one/two`,
     `task3_shoot_move_s[1..3]`, ...), dispatches exactly one `position_loop()` call against the
     matching waypoint table from `Motor.c`/`Servo.c` for that tick, then falls through to line
     crossing/IR-based Y-axis correction and `motor_speed_loop()`. **This is where you look to see
     which waypoint table actually fires for a given flag.**
   - `CCU60_CH1_IRQHandler` (100ms) ‚Äî LED heartbeat, `wheel_system_tick` counting, and a timed
     (`does_task3_start_to_count`, ~69 ticks) task3 rotation-then-handoff into
     `does_task2_dummy_move`.
   - `CCU61_CH0_IRQHandler` (1ms) ‚Äî integrates gyro rates into `wheel_asix.roll/pitch/yaw`
     (used everywhere else as the heading reference for radar/task gating), plus a
     `temp_test_flag`-gated timed spin used to hand off into `TASK2_SECOND_CALIBRATION`.
3. **`Cpu0_Main.c` main loop + `Core/Servo.c`** ‚Äî slow-cadence, self-guarded action sequences
   (each function checks its own precondition flag and returns immediately if not applicable):
   `TASK1_PICK_OBJECT_UP_SYN`/`TASK1_PUT_OBJECT_DOWN_SYN`, `TASK2_WATCH_DROP_WATER*`,
   `TASK2_PICK_AND_PUT_DROP_WATER_WORKFLOW` (arm/pump motion, blocking `delayms()` calls), plus
   the TFT status readout.

**Known gap**: `TASK3_WORKFLOW()` (`Core/Servo.c`) ‚Äî the function that actually fires the
shoot/reload sequence from a `worm_record_array` ‚Äî has no caller anywhere in the repository.
`worm_record_array` itself is populated by `UART0_RX_IRQHandler` mode 4, and the movement side
(`task3_shoot_move_s[1..3]`) is wired into the CCU6 scheduler, but nothing invokes
`TASK3_WORKFLOW()` to drive the shoot/reload half of task3 ‚Äî that's the piece still missing if
task3 doesn't fire end-to-end. Re-verify this by grepping for `TASK3_WORKFLOW(` before relying on
it, since this is exactly the kind of thing that gets wired up between commits.

## Working in this codebase

- File/variable/function comments mix Chinese and English and are stored as **GBK-encoded** text
  in several files (older files under `Core/`, the top-of-file banners in `Cpu0_Main.c`/
  `Cpu1_Main.c`, and the root `.txt` docs) alongside plain UTF-8 in newer files. If a file's
  Chinese comments render as mojibake when read directly, re-read it through
  `iconv -f GBK -t UTF-8` before trusting the text. When editing a GBK file, preserve its existing
  encoding unless you're deliberately normalizing it (mixed encodings can break the TASKING
  compiler's handling of string literals).
- Two-CPU shared state (e.g. `mutexCpu0InitIsOk`, `mutexCpu0TFTIsOk`, the TFT peripheral) uses
  `IfxCpu_mutexLock`/`acquireMutex`/`releaseMutex` plus plain `volatile` flags ‚Äî CPU1 currently
  does almost nothing, so most new work is CPU0-side; if you add CPU1 work that touches shared
  peripherals (TFT, UARTs), follow the existing mutex pattern rather than assuming exclusivity.
- Task state is tracked via loose global `volatile bool`/`volatile int` flags scattered across
  `Motor.h`/`Motor.c`, `Servo.h`/`Servo.c`, and `cheju.h`/`cheju.c` (e.g. `task2_start_correct`,
  `does_task_work_flow_start`, `task1_start_yaw_correction`) rather than a unified state machine
  struct, and they're read/written from both interrupt context (UART RX, CCU6 timers) and the main
  loop ‚Äî that's why they're `volatile`. When adding task behavior, follow the existing
  flag-driven convention (declare the flag where the related state lives, set it from the UART ISR
  or main loop, consume it from the CCU6 scheduler) instead of introducing a new pattern such as a
  central state struct.
- Pin assignments for this specific board are documented in
  `ÈæôÈÇ±TC2XX3XXÂçïÁâáÊú∫ËµÑÊ∫êÂàÜÈÖç.txt` (camera, encoders (GPT12), motors (GTM/ATOM0), servos
  (GTM/ATOM2), TFT/OLED, keys, LEDs, buzzer) ‚Äî check it before reassigning a peripheral pin, and
  keep `Libraries/pinmapper.pincfg` in sync with any change.
